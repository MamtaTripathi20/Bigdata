What This App Actually Does

A multi-mode music recommendation system with three distinct AI pipelines. Users authenticate, complete a "Vibe Check" onboarding quiz that seeds their taste profile, then receive personalized song recommendations. Songs can be played via YouTube's embed API (hidden iframe), added to queues, liked, and organized into playlists.

Critical Bug Investigation

HIGH PRIORITY
CRITICAL next() Button Is Completely Non-Functional PlayerContext.tsx · next()

The "Skip to next" button in PlayerBar calls next(), which does absolutely nothing. This is a self-cancelling guard bug — the function sets the "already fired" guard to true, then calls the handler whose first line returns early if the guard is true.

Root Cause
PlayerContext.tsx — next()
- songEndedFiredRef.current = true; // ← arms the guard
- handleSongEndedRef.current?.(); // ← then calls handler...
// Inside handleSongEndedRef.current:
- if (songEndedFiredRef.current) return; // ← ...which immediately returns!
Fix
PlayerContext.tsx — corrected next()
- const next = useCallback(() => {
- songEndedFiredRef.current = true;
- handleSongEndedRef.current?.();
- }, []);
+ const next = useCallback(() => {
+ // Skip the guard: manually force the next song
+ songEndedFiredRef.current = false; // clear guard so handler runs
+ handleSongEndedRef.current?.();
+ }, []);

Alternatively, extract queue-advance logic into its own function separate from the "song ended" guard, so manual skip bypasses the guard entirely.

CRITICAL Redis Connection Leaked on Every Recommendation Request routes/recommendations.py · get_recommendations()

_get_redis() creates a new async Redis client per request. In log_play_event it's properly closed with await redis.aclose(). But in get_recommendations — the hot path — the client is passed into pipelines and never closed. Under any real traffic, this exhausts the Redis connection pool.

Fix — Use a Context Manager or Dependency Injection
# Option A: async context manager (safest)
async def get_recommendations(...):
    async with aioredis.from_url(settings.REDIS_URL, decode_responses=False) as redis:
        if mode == RecommendationMode.ANALYTICS:
            return await AnalyticsPipeline(db, redis).recommend(...)
        ...

# Option B: FastAPI dependency with proper teardown
async def get_redis():
    r = aioredis.from_url(settings.REDIS_URL, decode_responses=False)
    try:
        yield r
    finally:
        await r.aclose()

# In route:
async def get_recommendations(..., redis = Depends(get_redis)):
    ...
CRITICAL Liking a Song Inflates play_count — Corrupts Recommendation Scores PlayerContext.tsx · toggleLike() + recommendations.py · log_play_event()

When a user clicks the ❤️ on any SongCard, toggleLike() calls recommendApi.logPlay({ song_id, liked }). The backend receives this as a PlayEvent with skipped=False (default), so it executes song.play_count += 1. Every like increases play_count even when the user never heard the song — which directly corrupts the Analytics popularity fallback and candidate pool ordering.

Additionally, likedSongs is an in-memory Set<number> in PlayerContext. On page refresh, all like state vanishes — but the backend already has a PlayHistory record with liked=true. The UI and DB are perpetually out of sync.

Fix 1 — Separate "like" from "play" on the backend
# In log_play_event — only increment play_count for actual plays
if not event.skipped and not event.liked:
    # A pure like event (song_id + liked=True, no other fields)
    # should not count as a play
    song.play_count += 1
Fix 2 — Hydrate likedSongs from backend on mount
// In PlayerProvider useEffect
useEffect(() => {
  const token = localStorage.getItem('access_token');
  if (!token) return;
  libraryApi.getLikedSongs()
    .then((r) => {
      const ids = r.data.songs.map((s: any) => s.song_id);
      setLikedSongs(new Set(ids));
    })
    .catch(() => {});
}, []);

HIGH Hybrid Fusion Weights Don't Sum to 1.0 — Scores Exceed 100% config.py · hybrid_pipeline.py · _fuse()

The config sets FUSION_ALPHA=0.6, FUSION_BETA=0.4, FUSION_GAMMA=0.1. Maximum possible score: 0.6 + 0.4 + 0.1 = 1.1. Songs with a diversity bonus will have score > 1.0, displayed in the UI as >100% via (song.score * 100).toFixed(0)%.

Fix — Normalize gamma or cap scores
# Option A: Reduce weights so they sum to ≤ 1.0
FUSION_ALPHA: float = 0.55
FUSION_BETA:  float = 0.35
FUSION_GAMMA: float = 0.10  # now sums to 1.0

# Option B: Normalize final scores after computing all
max_score = max(s for s, _ in scored) if scored else 1.0
scored = [(s / max(max_score, 1e-9), r) for s, r in scored]

HIGH Recommendations Table Is Never Written To db_models.py · all pipelines

The Recommendation DB model exists with full columns (user_id, song_id, mode, score, explanation). The User model has a recommendations relationship. But none of the three pipelines ever write to this table. Every recommendation request is a black hole — zero historical recommendation data is persisted for analysis, A/B testing, or model improvement.

Fix — Async background insert after response
# In get_recommendations(), after pipeline returns:
from fastapi import BackgroundTasks

async def _persist_recommendations(db, user_id, mode, recs):
    for rec in recs:
        db.add(Recommendation(
            user_id=user_id, song_id=rec.song_id,
            mode=mode.value, score=rec.score,
            explanation=rec.explanation,
        ))
    await db.commit()

# Add BackgroundTasks parameter and fire after return
background_tasks.add_task(_persist_recommendations, db, user_id, mode, result.recommendations)

Recommendation System Analysis

ML ANALYSIS

How Real Is the Analytics Pipeline?

The pipeline is legitimately real ML, not mock code. The FCM model at backend/models/fcm_model.pkl is only 2.1KB — suspicious. A real FCM fit on 10K songs × 9 features with 10 clusters should be ~50-100KB. This is likely a toy model trained on a small subset. The training code in train_fcm.py confirms this.

The collaborative filtering is a real item-item overlap SQL query finding similar users. The lookalike scorer combines cosine similarity and Jaccard on binarized features — a real technique. The weighted sum (0.30 + 0.40 + 0.30) is correct.

⚠️
The 5 MFCCs Are Weak Features
The feature vector uses only mfcc_1 through mfcc_5. Real music recommendation (like MSD research) uses 13-20 MFCC coefficients plus their deltas and delta-deltas. The current 5-MFCC + 4-continuous setup provides limited timbral discrimination. More importantly, the MSD dataset's actual MFCC values are often zero in the seed data as well.

How to Make Recommendations Actually Feel Personalized

  • Use the signal weights more aggressively in GenAI — if user has liked "sad slow ballads" in play history, inject that as a query modifier: "sad slow ballads like {artist}"
  • Track per-genre play ratios — if 70% of plays are Hip-Hop, bias iTunes searches toward Hip-Hop queries proportionally
  • Use the vibe check persona in GenAI queries — "Hype Machine" users → energy queries; "Deep Listener" users → ambient/jazz queries
  • Collaborative filtering is currently weak — only counts overlap, not weighted by signal. A skipped overlapping song should count less than a liked one. Weight CF by SIGNAL_LIKE / SIGNAL_PLAY ratio
  • Diversify beyond play_count ordering for candidates — the 200-song pool is ordered by play_count DESC. Popular songs dominate. Add a recency slot: include some recently-added songs regardless of play_count

Performance Analysis

OPTIMIZATION

Backend Performance Issues

N+1 Query Risk
_save_song_to_db()
4 separate SQL queries per song (check existing, genre lookup/insert, artist lookup/insert, song insert). For 45 candidates → up to 180 queries per request.
Collaborative Filter
Uncached
The CF query runs a full join across play_history on every recommendation request. Should be cached per user_id with a 5min TTL alongside the user vector.
Embedding Warmup
~78 seconds
Model warmup in lifespan is correct. But run_in_executor with the default thread pool can block other startup tasks. Use a dedicated ThreadPoolExecutor.
Ticker Interval
500ms × ∞
PlayerContext runs a 500ms setInterval indefinitely while isPlaying. Low overhead, but not cleaned up if PlayerProvider unmounts. Add cleanup in useEffect return.

Fix: Batch Song Saves in GenAI Pipeline

# Instead of N queries per song, batch the lookup
itunes_ids = [c['itunes_id'] for c in candidates]
existing = await db.execute(
    text("SELECT id, mb_id FROM songs WHERE mb_id = ANY(:ids)"),
    {"ids": itunes_ids}
)
existing_map = {row.mb_id: row.id for row in existing.fetchall()}

# Only process songs not already in DB
new_candidates = [c for c in candidates if c['itunes_id'] not in existing_map]

Frontend Performance

  • ComparisonDashboard fires 3 simultaneous API requests (via useAllModes) — this is by design and acceptable. React Query's staleTime prevents refetch on re-render.
  • SongCard cover images have no lazy loading. Add loading="lazy" to <img> tags to defer off-screen images.
  • YouTube iframe persists across song changes (same iframe, different src). This forces a full iframe reload each song. Consider using the YT IFrame Player API properly with loadVideoById() instead of changing the src.

Docker & Dev Setup

DEVOPS

Issues Found

HIGH Frontend NEXT_PUBLIC_API_URL Points to localhost — Breaks in CI/Staging docker-compose.yml

The frontend container has NEXT_PUBLIC_API_URL=http://localhost:8000. In dev mode (npm run dev), NEXT_PUBLIC_ variables are available at runtime via process.env. Since all API calls are client-side (axios), localhost:8000 is resolved by the browser, not the container — so it works when port 8000 is exposed. However, any SSR fetch (getServerSideProps etc.) or CI environment would break.

Fix
docker-compose.yml
- - NEXT_PUBLIC_API_URL=http://localhost:8000
+ - NEXT_PUBLIC_API_URL=http://localhost:8000 # browser → host mapping, correct for dev
+ # For SSR: also set INTERNAL_API_URL=http://backend:8000
HIGH Backend Volume Mount Overrides Docker-Built Models docker-compose.yml · backend service

The backend container has volumes: - ./backend:/app. This mounts the host's ./backend over the container's /app. The fcm_model.pkl is in backend/models/ on host, so it works. But the COPY . . instruction in the Dockerfile is effectively ignored at runtime — all files come from the volume mount. If the model file is missing on the host, the container silently falls back to live FCM.

The --reload flag in the compose command also means any code edit triggers a full reload — the 78-second embedding model warmup runs every time a Python file is saved during development.

Fix — Separate model volume from code reload
# docker-compose.yml — mount code but keep models from image
backend:
  volumes:
    - ./backend/app:/app/app          # only hot-reload app code
    - backend_models:/app/models      # models persist from image
  command: uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
           --reload-dir /app/app      # only watch app/ for reloads

What Works Well in Docker Setup

  • Health checks on both postgres and redis with depends_on condition: service_healthy — correct startup ordering
  • Torch CPU-only installed from PyTorch index before requirements.txt — avoids CUDA download, saves ~2GB
  • Redis LRU eviction policy (allkeys-lru) with 256MB limit — prevents OOM
  • Named volumes for postgres_data and redis_data — data persists across container restarts
⚠️
Stray Files at Project Root
album_page.tsx, artist_page.tsx, library.py, PlayerContext.tsx, artist_album_routes.py all exist at the repo root — duplicates of files in frontend/src/ and backend/app/. These are development leftovers that will cause confusion and may hide the actual running code. Delete them.
